## 2026년 프론트엔드 생존 전략: React Hook Form 베스트 프랙티스, Biome 커스텀 룰로 강제하기!

### 🚀 소개: 왜 이 주제가 중요한가요?

React Hook Form (RHF)은 React 애플리케이션에서 폼을 다루는 가장 인기 있고 효율적인 라이브러리 중 하나입니다. 하지만 강력한 만큼, 잘못 사용하면 불필요한 리렌더링이나 예측하기 어려운 동작으로 이어질 수 있어 성능 저하의 주범이 되기도 합니다.

오늘 우리는 RHF의 잠재적인 성능 저하 요소를 사전에 방지하고, 코드의 일관성과 품질을 높이는 Biome 커스텀 룰 활용법을 깊이 있게 다뤄볼 예정입니다. 이는 개발 팀 전체의 생산성을 향상시키고, 장기적으로 안정적인 애플리케이션을 구축하는 데 필수적인 전략이 될 것입니다.

### 💡 핵심 개념 설명: React Hook Form & Biome & GritQL

#### React Hook Form (RHF)의 흔한 함정

RHF는 폼 상태 관리 및 유효성 검사를 위한 강력한 API를 제공하지만, 몇 가지 사용 패턴은 주의가 필요합니다:

1.  **`watch()`의 무분별한 사용:** `watch()` 함수를 인자 없이 호출하면 폼의 모든 필드를 구독하게 되어, 어떤 필드든 변경될 때마다 컴포넌트 전체가 리렌더링됩니다. 이는 불필요한 성능 저하의 주된 원인이 됩니다. 필요한 특정 필드만 구독하거나, `useWatch` 훅을 활용하는 것이 좋습니다.
2.  **`setValue()`의 부적절한 활용:** `setValue()`는 특정 필드의 값을 설정하는 데 유용하지만, `shouldValidate` 옵션 없이 사용하면 유효성 검사가 누락될 수 있고, `shouldDirty` 옵션 없이 사용하면 폼의 `isDirty` 상태가 업데이트되지 않아 예상치 못한 동작을 유발할 수 있습니다.

#### Biome: 차세대 코드 툴체인

Biome은 Rome의 정신을 이어받은 차세대 코드 툴체인으로, 포맷터, 린터, 번들러, 테스터 등 다양한 기능을 하나의 도구에서 제공하여 개발 워크플로우를 간소화합니다. 특히 강력한 점은 바로 **커스텀 룰**을 정의할 수 있다는 것입니다.

#### GritQL: Biome 커스텀 룰의 핵심

Biome은 GritQL이라는 강력한 패턴 매칭 언어를 사용하여 커스텀 린트 룰을 정의할 수 있게 합니다. GritQL은 코드의 추상 구문 트리(AST)를 기반으로 특정 코드 패턴을 검색하고, 필요에 따라 코드를 변환할 수도 있는 기능을 제공합니다. 이를 통해 우리는 RHF의 베스트 프랙티스를 강제하는 맞춤형 룰을 생성할 수 있습니다.

### 🛠️ 실제 코드 예시: `watch()` 오용 방지 룰 만들기

이제 `watch()`를 인자 없이 호출하여 불필요한 리렌더링을 유발하는 패턴을 감지하고 경고하는 Biome 커스텀 룰을 만들어 보겠습니다.

#### 1. 문제 상황: `watch()` 오용 예시 (`components/BadForm.tsx`)

아래 코드는 `watch()`를 인자 없이 호출하여 모든 필드의 변경을 구독하고, 이로 인해 불필요하게 컴포넌트가 리렌더링되는 상황을 보여줍니다.

```tsx
// components/BadForm.tsx
import React from 'react';
import { useForm } from 'react-hook-form';

type FormData = {
  firstName: string;
  lastName: string;
  email: string;
};

export default function BadForm() {
  const { register, handleSubmit, watch } = useForm<FormData>();
  const allFields = watch(); // 🚨 문제: 인자 없이 watch()를 호출하여 모든 필드를 구독

  const onSubmit = (data: FormData) => console.log(data);

  // 이 컴포넌트는 firstName, lastName, email 중 어떤 필드라도 변경되면 리렌더링됩니다.
  console.log('BadForm 리렌더링!', allFields);

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <label>
        First Name:
        <input {...register("firstName")} placeholder="First Name" />
      </label>
      <label>
        Last Name:
        <input {...register("lastName")} placeholder="Last Name" />
      </label>
      <label>
        Email:
        <input {...register("email")} placeholder="Email" />
      </label>
      <p>현재 First Name: {allFields.firstName}</p> {/* 특정 필드만 필요할 때도 전체 구독 */}
      <button type="submit">Submit</button>
    </form>
  );
}
```

#### 2. Biome 커스텀 룰 정의: `noUnnecessaryWatch.grit`

`rules/noUnnecessaryWatch.grit` 파일을 생성하고 아래 GritQL 룰을 작성합니다. 이 룰은 `useForm`에서 `watch` 함수를 가져와 인자 없이 호출하는 패턴을 찾아냅니다.

```grit
// rules/noUnnecessaryWatch.grit
pattern no_unnecessary_watch() {
    language js

    // `useForm` 훅 호출을 찾습니다.
    `useForm($params)` as $use_form_call

    // `useForm`의 반환값에서 `watch` 함수를 구조 분해 할당으로 가져오는 부분을 찾습니다.
    `const { ..., watch: $watch_fn, ... } = $use_form_call`

    // 이제 `$watch_fn`이 인자 없이 호출되는 모든 경우를 찾습니다.
    `$watch_fn()` as $problematic_watch_call

    // 이 패턴이 발견되면 경고 메시지를 출력합니다.
    where {
        message => "React Hook Form의 `watch()`를 인자 없이 호출하면 불필요한 리렌더링이 발생할 수 있습니다. 특정 필드만 구독하거나 `useWatch`를 사용하세요."
    }
}
```

#### 3. `biome.json` 설정 업데이트

`biome.json` 파일에 커스텀 룰을 등록하여 Biome이 이를 인식하도록 합니다. `linter.rules.custom` 섹션에 우리가 만든 룰을 추가합니다.

```json
// biome.json
{
  "$schema": "https://biomejs.dev/schemas/1.6.3/schema.json",
  "organizeImports": {
    "enabled": true
  },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true,
      "nursery": {
        "noUnusedVariables": "error"
      },
      "custom": {
        "noUnnecessaryWatch": "error" // 여기에 커스텀 룰 추가
      }
    }
  },
  "files": {
    "ignore": [
      "node_modules",
      "dist"
    ]
  },
  // Biome이 커스텀 룰 파일을 찾을 수 있도록 설정
  "vcs": {
    "root": "./"
  }
}
```
*Note: Biome 1.6.x 버전부터 커스텀 룰은 `vcs.root` 설정이 필요할 수 있습니다. 또한, 커스텀 룰 파일은 프로젝트 루트의 `rules` 디렉토리 안에 있어야 합니다.*

#### 4. Biome 실행 및 결과 확인

이제 터미널에서 `biome lint .` 명령어를 실행하면, `BadForm.tsx` 파일에서 `watch()` 오용 패턴을 감지하여 경고 메시지를 출력할 것입니다.

```bash
$ biome lint .
components/BadForm.tsx:16:20 error React Hook Form의 `watch()`를 인자 없이 호출하면 불필요한 리렌더링이 발생할 수 있습니다. 특정 필드만 구독하거나 `useWatch`를 사용하세요.
```

#### 5. 개선된 코드 예시: `watch()` 올바른 사용 (`components/GoodForm.tsx`)

`watch()`를 필요한 경우에만 특정 필드와 함께 사용하거나, `useWatch` 훅을 활용하여 컴포넌트 전체의 리렌더링을 방지합니다.

```tsx
// components/GoodForm.tsx
import React from 'react';
import { useForm } from 'react-hook-form';
import { useWatch } from 'react-hook-form'; // useWatch 훅 임포트

type FormData = {
  firstName: string;
  lastName: string;
  email: string;
};

export default function GoodForm() {
  const { register, handleSubmit, control } = useForm<FormData>();
  // 💡 필요한 특정 필드만 구독하거나, useWatch 훅을 사용합니다.
  const firstName = useWatch({
    control,
    name: "firstName",
    // defaultValue: "홍길동" // 초기값이 필요한 경우
  });

  const onSubmit = (data: FormData) => console.log(data);

  // 이 컴포넌트는 firstName이 변경될 때만 리렌더링됩니다.
  console.log('GoodForm 리렌더링! firstName:', firstName);

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <label>
        First Name:
        <input {...register("firstName")} placeholder="First Name" />
      </label>
      <label>
        Last Name:
        <input {...register("lastName")} placeholder="Last Name" />
      </label>
      <label>
        Email:
        <input {...register("email")} placeholder="Email" />
      </label>
      <p>안녕하세요, {firstName}님!</p> {/* firstName을 활용 */}
      <button type="submit">Submit</button>
    </form>
  );
}
```

### 🎯 실무 적용 팁

1.  **CI/CD 파이프라인 통합:** Biome 린팅을 CI/CD 파이프라인에 통합하여, 베스트 프랙티스를 위반하는 코드가 프로덕션으로 배포되는 것을 사전에 방지하세요. GitHub Actions, GitLab CI 등에서 `biome lint` 명령어를 실행하도록 설정할 수 있습니다.
2.  **점진적인 적용:** 모든 베스트 프랙티스를 한 번에 강제하기보다는, 팀의 우선순위에 따라 가장 중요하고 영향력 있는 룰부터 점진적으로 적용해나가세요.
3.  **팀원 교육 및 논의:** 새로 추가된 룰과 그 배경에 대해 팀원들과 공유하고 논의하는 시간을 가지세요. 단순히 에러를 발생시키는 것을 넘어, 왜 해당 룰이 필요한지 이해시키는 것이 중요합니다.
4.  **TypeScript와의 시너지:** TypeScript의 타입 안전성과 Biome의 린팅 룰을 함께 활용하면 코드의 견고함과 예측 가능성을 극대화할 수 있습니다.
5.  **더 복잡한 룰 탐색:** `setValue`의 `shouldValidate` 또는 `shouldDirty` 옵션 누락을 감지하는 룰 등, 프로젝트의 특성에 맞는 다양한 커스텀 룰을 GritQL로 구현해 보세요.

### 📝 마무리: 코드 품질의 새로운 지평을 열다

React Hook Form은 프론트엔드 개발자에게 강력한 도구이지만, 그 잠재력을 최대한 발휘하기 위해서는 올바른 사용법이 필수적입니다. Biome의 커스텀 룰 기능을 활용하면, 이러한 베스트 프랙티스를 코드 레벨에서 강제하고, 잠재적인 버그와 성능 문제를 사전에 방지할 수 있습니다.

이는 단순히 코드를 깨끗하게 만드는 것을 넘어, 개발 팀 전체의 생산성을 높이고, 유지보수가 용이하며 안정적인 애플리케이션을 구축하는 데 결정적인 역할을 할 것입니다. 여러분의 프로젝트에서도 Biome과 GritQL을 적극적으로 활용하여 코드 품질의 새로운 지평을 열어보시길 강력히 추천합니다!
