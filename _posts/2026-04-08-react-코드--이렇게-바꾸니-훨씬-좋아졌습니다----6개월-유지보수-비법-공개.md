대규모 React 애플리케이션을 개발하고 유지보수하는 것은 끊임없는 도전입니다. 특히 출시 후 6개월이 지나도 복잡성 없이 깔끔하게 유지되는 코드는 개발팀의 꿈이죠. 오늘 우리는 '무엇을 멈춰야 더 나은 코드를 만들 수 있는지'에 대한 깊이 있는 통찰을 얻게 될 것입니다.

## 왜 이 주제가 중요할까요?

프로젝트 초기에는 빠르게 기능을 구현하는 것이 중요하지만, 시간이 지날수록 코드의 복잡성은 기하급수적으로 증가하고 유지보수 비용은 눈덩이처럼 불어납니다. Muhammed Sodiq의 글은 복잡한 마케팅 자동화 플랫폼을 성공적으로 운영하며 얻은 실질적인 교훈을 공유합니다. 그의 경험을 통해 우리는 단순히 기능을 추가하는 것을 넘어, 미래를 위한 견고하고 유지보수 가능한 React 코드를 작성하는 방법을 배울 수 있습니다.

## 핵심 개념: '멈춤'의 미학

Muhammed Sodiq는 코드의 품질과 유지보수성을 높이기 위해 다음과 같은 관행들을 '멈췄다'고 이야기합니다. 이는 단순히 안티패턴을 피하는 것을 넘어, 더 의도적이고 효율적인 개발 방식을 채택하는 것을 의미합니다.

1.  **지나친 Prop Drilling:** 컴포넌트 트리를 깊게 통과하는 Prop들은 코드를 읽기 어렵게 만들고 리팩토링을 어렵게 합니다.
2.  **모든 것을 전역 상태로 관리하려는 시도:** 불필요한 전역 상태는 코드의 복잡성을 높이고 예측 불가능한 버그를 유발합니다.
3.  **책임이 모호한 거대 컴포넌트:** 하나의 컴포넌트가 너무 많은 역할을 담당하면 재사용성이 떨어지고 테스트가 어려워집니다.
4.  **불필요한 리렌더링에 대한 무관심:** 성능 저하의 주범인 불필요한 리렌더링을 방치하는 것은 사용자 경험에 악영향을 줍니다.
5.  **어설픈 TypeScript 사용:** TypeScript를 단순히 `any` 타입으로 도배하거나 제대로 활용하지 않으면, 그 장점을 잃고 오히려 개발 속도만 저해합니다.

그렇다면, 이러한 관행들을 멈추고 어떻게 더 나은 코드를 만들 수 있을까요?

### 1. Prop Drilling 대신 Context API 또는 전역 상태 관리 라이브러리 활용

깊은 컴포넌트 계층에서 데이터를 전달해야 할 때, Prop Drilling은 코드를 지저분하게 만듭니다. 이때 React Context API나 Zustand, Recoil, Redux 같은 상태 관리 라이브러리를 활용하여 필요한 컴포넌트에만 데이터를 직접 주입할 수 있습니다.

**나쁜 예시 (Prop Drilling):**

```tsx
// components/ParentComponent.tsx
import React, { useState } from 'react';
import ChildComponent from './ChildComponent';

interface ParentProps {
  initialCount: number;
}

const ParentComponent: React.FC<ParentProps> = ({ initialCount }) => {
  const [count, setCount] = useState(initialCount);

  return (
    <div>
      <h1>Parent: {count}</h1>
      {/* count와 setCount를 Child를 거쳐 Grandchild까지 전달 */}
      <ChildComponent count={count} setCount={setCount} />
    </div>
  );
};

export default ParentComponent;
```

```tsx
// components/GrandchildComponent.tsx
// ChildComponent를 거쳐온 Props를 사용
import React from 'react';

interface GrandchildProps {
  count: number;
  setCount: React.Dispatch<React.SetStateAction<number>>;
}

const GrandchildComponent: React.FC<GrandchildProps> = ({ count, setCount }) => {
  return (
    <div>
      <h3>Grandchild: {count}</h3>
      <button onClick={() => setCount(prev => prev + 1)}>Increment</button>
    </div>
  );
};

export default GrandchildComponent;
```

**좋은 예시 (Context API 활용):**

```tsx
// contexts/CounterContext.tsx (TypeScript 적용)
import React, { createContext, useContext, useState, ReactNode } from 'react';

interface CounterContextType {
  count: number;
  setCount: React.Dispatch<React.SetStateAction<number>>;
}

// undefined를 허용하고, 사용 시 체크하여 런타임 에러 방지
const CounterContext = createContext<CounterContextType | undefined>(undefined);

export const CounterProvider: React.FC<{ children: ReactNode; initialCount?: number }> = ({ children, initialCount = 0 }) => {
  const [count, setCount] = useState(initialCount);
  return (
    <CounterContext.Provider value={{ count, setCount }}>
      {children}
    </CounterContext.Provider>
  );
};

export const useCounter = () => {
  const context = useContext(CounterContext);
  if (context === undefined) {
    throw new Error('useCounter must be used within a CounterProvider');
  }
  return context;
};
```

```tsx
// components/GrandchildComponentWithContext.tsx
import React from 'react';
import { useCounter } from '../contexts/CounterContext'; // Context에서 직접 값을 가져옴

const GrandchildComponentWithContext: React.FC = () => {
  const { count, setCount } = useCounter(); // Prop Drilling 없이 직접 접근

  return (
    <div>
      <h3>Grandchild (with Context): {count}</h3>
      <button onClick={() => setCount(prev => prev + 1)}>Increment</button>
    </div>
  );
};

export default GrandchildComponentWithContext;
```

```tsx
// pages/index.tsx (Next.js 페이지에서 Provider 사용 예시)
import { CounterProvider } from '../contexts/CounterContext';
import ParentComponentWithContext from '../components/ParentComponentWithContext';

const HomePage: React.FC = () => {
  return (
    <CounterProvider initialCount={0}>
      <ParentComponentWithContext /> {/* ParentComponent는 이제 count를 prop으로 받지 않아도 됩니다. */}
    </CounterProvider>
  );
};

export default HomePage;
```

이처럼 Context API를 사용하면 `GrandchildComponent`는 `Parent`나 `Child`로부터 Prop을 전달받을 필요 없이 `useCounter` 훅을 통해 필요한 상태에 직접 접근할 수 있습니다. 이는 코드의 가독성을 높이고 컴포넌트 간의 결합도를 낮춥니다.

### 2. 컴포넌트 분리와 관심사 분리 (SoC)

하나의 컴포넌트가 너무 많은 UI와 로직을 포함하면 유지보수가 어려워집니다. 컴포넌트를 작고 재사용 가능한 단위로 분리하고, 각 컴포넌트가 하나의 명확한 책임만 가지도록 설계하세요. (예: 데이터 페칭 로직, UI 렌더링 로직, 비즈니스 로직 분리)

### 3. 현명한 성능 최적화 (feat. `React.memo`, `useMemo`, `useCallback`)

모든 곳에 `memo`를 적용하는 것은 오히려 오버헤드를 발생시킬 수 있습니다. 하지만, 비용이 많이 드는 계산, 복잡한 UI 렌더링, 자주 리렌더링 되는 리스트 등 성능 병목 지점에서 `React.memo`, `useMemo`, `useCallback`을 전략적으로 사용하면 불필요한 리렌더링을 줄여 애플리케이션의 반응성을 크게 향상시킬 수 있습니다.

### 4. TypeScript를 '제대로' 활용하기

TypeScript는 단순한 타입 힌트를 넘어, 코드의 계약을 명확히 하고 잠재적인 런타임 오류를 컴파일 시점에 잡아주는 강력한 도구입니다. 제네릭, 유틸리티 타입, 조건부 타입 등을 적극 활용하여 더 견고하고 예측 가능한 코드를 작성하세요. 이는 협업 효율성을 높이고 장기적인 유지보수 비용을 절감하는 데 필수적입니다.

## 실무 적용 팁

*   **코드 리뷰 활성화:** 동료들과 코드 리뷰를 통해 비효율적인 패턴을 조기에 발견하고 개선하는 문화를 만드세요.
*   **린터(ESLint)와 포매터(Prettier) 설정:** 일관된 코드 스타일을 유지하고 잠재적인 문제를 자동으로 감지하도록 설정하세요.
*   **작은 단위부터 리팩토링 시작:** 한 번에 모든 것을 바꾸려 하지 말고, 새로운 기능을 개발할 때마다 주변 코드를 조금씩 개선해 나가세요.
*   **테스트 코드 작성:** 버그를 방지하고 리팩토링 시 안정성을 확보하기 위해 테스트 코드를 충실히 작성하세요.
