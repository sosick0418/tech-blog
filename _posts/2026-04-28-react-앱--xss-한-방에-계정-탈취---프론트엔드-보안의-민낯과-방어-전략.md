## 소개

우리는 React로 멋진 사용자 경험을 만들어내지만, 혹시 그 아름다운 UI 뒤에 숨겨진 치명적인 보안 취약점은 없는지 생각해보셨나요? 오늘 다룰 주제는 바로 **XSS(Cross-Site Scripting)** 공격입니다. "백엔드에서 다 알아서 해주겠지"라는 안일한 생각은 금물입니다. 여러분의 React 애플리케이션이 단 한 번의 XSS 공격으로 사용자 계정 탈취라는 심각한 위협에 노출될 수 있다는 사실, 알고 계셨나요? 프론트엔드 개발자로서 XSS의 위험성을 정확히 인지하고, 강력한 방어 전략을 구축하는 것은 선택이 아닌 필수입니다.

## 본문

### 핵심 개념 설명: XSS, 왜 그렇게 위험한가?

XSS는 공격자가 악성 스크립트를 웹사이트에 주입하여 다른 사용자들의 브라우저에서 실행되도록 하는 웹 취약점입니다. 단순히 페이지를 망가뜨리는 수준을 넘어, 세션 쿠키 탈취, 로컬 스토리지 데이터 접근, 심지어 사용자 인터페이스를 조작하여 가짜 로그인 폼을 띄우는 등 상상 이상의 피해를 입힐 수 있습니다. 특히, `HttpOnly` 플래그가 설정되지 않은 쿠키나 `localStorage`, `sessionStorage`에 저장된 인증 토큰은 XSS 공격에 매우 취약하며, 이는 곧 **사용자 계정 탈취**로 이어질 수 있습니다.

React 애플리케이션의 경우, 주로 사용자로부터 입력받은 데이터를 적절히 처리하지 않고 DOM에 직접 렌더링할 때 XSS 취약점이 발생합니다. 특히 `dangerouslySetInnerHTML`과 같은 기능을 사용할 때 각별한 주의가 필요합니다. React는 기본적으로 JSX 내의 문자열을 자동으로 이스케이프하여 XSS를 방어하지만, 개발자가 의도적으로 이 보호막을 해제하거나 우회할 때 문제가 발생합니다.

### 실제 코드 예시: 안전한 React 렌더링

#### 🚨 XSS 취약 코드 (절대 따라 하지 마세요!)

다음은 사용자 입력(`comment.text`)을 `dangerouslySetInnerHTML`을 사용하여 직접 렌더링하는 예시입니다. 만약 `comment.text`에 `<script>alert('XSS!');</script>`와 같은 악성 코드가 포함되어 있다면, 해당 스크립트가 실행될 것입니다.

```tsx
// src/components/VulnerableComment.tsx
import React from 'react';

interface CommentProps {
  comment: {
    id: string;
    text: string;
    author: string;
  };
}

function VulnerableComment({ comment }: CommentProps) {
  return (
    <div className="comment-item">
      <h4>{comment.author}</h4>
      {/* 🚨 경고: dangerouslySetInnerHTML 사용 시 사용자 입력에 대한 검증/살균이 없으면 XSS 취약! */}
      <div dangerouslySetInnerHTML={{ __html: comment.text }} />
    </div>
  );
}

export default VulnerableComment;

/*
  예상되는 악성 입력:
  comment.text = "<img src='x' onerror='alert(\"XSS 공격 성공!\"); document.cookie=\"stolen=\" + document.cookie;'>"
*/
```

#### ✅ 안전한 React 렌더링 (React의 기본 방어)

React는 JSX 내부의 문자열을 자동으로 이스케이프(Escape)합니다. 따라서 대부분의 경우, 별도의 조치 없이 사용자 입력을 `{}` 안에 넣어 렌더링하는 것만으로 XSS를 방어할 수 있습니다.

```tsx
// src/components/SecureComment.tsx
import React from 'react';

interface CommentProps {
  comment: {
    id: string;
    text: string;
    author: string;
  };
}

function SecureComment({ comment }: CommentProps) {
  return (
    <div className="comment-item">
      <h4>{comment.author}</h4>
      {/* ✅ React는 `{}` 내부의 문자열을 자동으로 이스케이프하여 XSS를 방어합니다. */}
      <div>{comment.text}</div>
    </div>
  );
}

export default SecureComment;

/*
  위의 악성 입력이 SecureComment에 전달되면:
  <div>&lt;img src=&#x27;x&#x27; onerror=&#x27;alert(&quot;XSS 공격 성공!&quot;); ...&#x27;&gt;</div>
  와 같이 안전하게 렌더링됩니다.
*/
```

#### ✨ `dangerouslySetInnerHTML` 사용 시 안전하게 처리하는 방법 (DOMPurify 활용)

하지만 특정 상황(예: 마크다운 렌더링, 특정 HTML 태그 허용)에서 `dangerouslySetInnerHTML`을 사용해야 한다면, **반드시 신뢰할 수 있는 HTML 살균(Sanitization) 라이브러리를 사용해야 합니다.** `DOMPurify`는 클라이언트 사이드에서 HTML을 안전하게 살균하는 데 널리 사용되는 라이브러리입니다.

```tsx
// src/components/SanitizedHtmlDisplay.tsx
import React, { useMemo } from 'react';
import DOMPurify from 'dompurify'; // npm install dompurify

interface SanitizedHtmlProps {
  htmlContent: string;
}

function SanitizedHtmlDisplay({ htmlContent }: SanitizedHtmlProps) {
  // useMemo를 사용하여 htmlContent가 변경될 때만 살균 작업을 수행합니다.
  const cleanHtml = useMemo(() => {
    return DOMPurify.sanitize(htmlContent, { USE_PROFILES: { html: true } });
  }, [htmlContent]);

  return (
    <div
      // ✅ 살균된 HTML만 dangerouslySetInnerHTML에 전달합니다.
      dangerouslySetInnerHTML={{ __html: cleanHtml }}
    />
  );
}

export default SanitizedHtmlDisplay;

/*
  악성 입력:
  htmlContent = "<h1>안녕</h1><script>alert('XSS!');</script><a href='javascript:alert(\"또 XSS!\")'>링크</a>"

  DOMPurify.sanitize 후:
  "<h1>안녕</h1><a>링크</a>"
  스크립트와 위험한 href 속성이 제거됩니다.
*/
```

### 실무 적용 팁

1.  **사용자 입력은 항상 살균(Sanitize)하세요:** 서버 사이드뿐만 아니라 클라이언트 사이드에서도 사용자 입력을 신뢰하지 마세요. `DOMPurify`와 같은 라이브러리를 사용하여 HTML 태그, 속성 등을 필터링하고 안전한 것만 허용해야 합니다.
2.  **`dangerouslySetInnerHTML` 사용은 최후의 수단으로:** 꼭 필요한 경우가 아니라면 이 기능을 사용하지 마세요. React의 기본 렌더링 방식만으로도 대부분의 XSS를 방어할 수 있습니다. 사용해야 한다면 위 예시처럼 반드시 살균 과정을 거쳐야 합니다.
3.  **HTTP-Only 쿠키를 사용하세요:** 인증 토큰과 같은 민감한 정보는 `HttpOnly` 플래그가 설정된 쿠키에 저장하여 클라이언트 사이드 스크립트가 접근하지 못하도록 하세요. 이는 XSS 공격으로 인한 토큰 탈취를 효과적으로 방어할 수 있습니다.
4.  **Content Security Policy (CSP)를 구현하세요:** CSP는 브라우저가 로드할 수 있는 리소스(스크립트, 스타일시트, 이미지 등)의 출처를 지정하여 XSS 공격을 포함한 다양한 콘텐츠 주입 공격을 완화하는 강력한 보안 계층입니다. `nonce`나 `hash`를 사용하여 인라인 스크립트를 제한하는 것이 중요합니다.
5.  **보안 라이브러리 및 프레임워크 최신 버전 유지:** React, Next.js 등 사용하는 모든 라이브러리와 프레임워크를 최신 버전으로 유지하여 알려진 취약점을 패치하세요.
6.  **정기적인 보안 감사 및 코드 리뷰:** 팀 내에서 보안에 대한 인식을 높이고, 코드 리뷰 시 XSS 취약점 가능성을 항상 염두에 두어야 합니다. 자동화된 보안 스캐너를 활용하는 것도 좋은 방법입니다.

## 마무리

오늘 우리는 React 애플리케이션에서 XSS 공격이 얼마나 치명적일 수 있는지, 그리고 이를 어떻게 효과적으로 방어할 수 있는지 살펴보았습니다. XSS는 단순히 불편함을 주는 버그가 아니라, 사용자 계정 탈취와 같은 심각한 보안 사고로 이어질 수 있는 고위험 취약점입니다.

프론트엔드 개발자로서 우리는 아름다운 UI를 만드는 것뿐만 아니라, 사용자의 데이터를 안전하게 보호할 책임이 있습니다. React의 기본 방어 기능을 신뢰하되, `dangerouslySetInnerHTML`과 같은 특정 기능을 사용할 때는 반드시 `DOMPurify`와 같은 살균 라이브러리를 활용하고, `HttpOnly` 쿠키 및 CSP와 같은 추가적인 방어막을 구축하여 다층적인 보안 전략을 세워야 합니다.

이제 여러분의 React 애플리케이션 코드를 다시 한번 들여다보며 혹시 놓친 보안 취약점은 없는지 점검해 볼 시간입니다. 다음 학습 방향으로는 CSP 구성에 대해 더 깊이 파고들고, OWASP Top 10과 같은 웹 보안 표준에 대해 학습해보시는 것을 추천합니다. 우리 모두 안전한 웹 환경을 만드는 데 기여합시다!
