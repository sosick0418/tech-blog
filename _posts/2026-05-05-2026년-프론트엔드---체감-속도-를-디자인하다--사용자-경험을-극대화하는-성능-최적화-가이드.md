## 🚀 서론: 왜 프론트엔드 성능에 집중해야 하는가?

오늘날 웹은 단순히 정보를 전달하는 도구를 넘어, 풍부한 상호작용과 다이내믹한 경험을 제공하는 플랫폼으로 진화했습니다. 사용자들은 단 0.1초의 지연에도 민감하게 반응하며, 이는 이탈률 증가, 전환율 하락, 심지어 브랜드 이미지 손상으로 이어질 수 있습니다. 2026년의 프론트엔드 개발자라면, 이제 '기능 구현'을 넘어 '체감 속도'를 디자인하는 것이 핵심 역량이 되었습니다. ## 💡 핵심 개념 파헤치기: 레이아웃 스래싱, INP, will-change, 패시브 리스너

프론트엔드 성능 최적화의 첫걸음은 문제의 원인을 정확히 이해하는 것입니다. 다음 네 가지 핵심 개념을 통해 우리 웹사이트의 '병목 현상'을 진단하고 개선할 수 있습니다.

### 1. 레이아웃 스래싱 (Layout Thrashing)

**핵심 개념:** 레이아웃 스래싱은 브라우저가 DOM(Document Object Model)의 레이아웃을 계산하고(읽기), 변경하고(쓰기), 다시 계산하는(읽기) 작업을 짧은 시간 안에 번갈아 가며 반복할 때 발생합니다. 예를 들어, 요소의 `clientWidth`를 읽은 직후 `width`를 변경하면, 브라우저는 변경된 `width`를 적용하기 위해 강제로 레이아웃을 다시 계산해야 합니다. 이러한 강제 동기식 레이아웃은 렌더링 파이프라인을 멈추게 하여 성능 저하를 유발합니다.

**실제 코드 예시 (React/TypeScript):**

아래 예시는 레이아웃 스래싱이 발생하는 비효율적인 패턴과 이를 개선하는 방법을 보여줍니다.

```tsx
// components/LayoutThrashingExample.tsx
import React, { useEffect, useRef, useState } from 'react';

const LayoutThrashingExample: React.FC = () => {
  const boxRef = useRef<HTMLDivElement>(null);
  const [trigger, setTrigger] = useState(0);

  // 🔴 비효율적인 예시: 레이아웃 스래싱 유발 가능성
  // DOM 읽기(clientWidth)와 쓰기(style.height 변경)가 같은 프레임 내에서 번갈아 발생
  useEffect(() => {
    if (boxRef.current && trigger > 0) {
      const initialWidth = boxRef.current.clientWidth; // 1. DOM 읽기
      boxRef.current.style.height = `${Math.random() * 50 + 50}px`; // 2. DOM 쓰기 (레이아웃 무효화)
      const finalWidth = boxRef.current.clientWidth; // 3. DOM 읽기 (강제 레이아웃 재계산 발생)
      console.log(`🔴 Layout Thrashing: ${initialWidth} -> ${finalWidth}`);
    }
  }, [trigger]); // trigger 상태가 변할 때마다 실행

  // ✨ 개선된 예시: requestAnimationFrame을 활용한 배치 처리
  // DOM 읽기와 쓰기를 분리하여 레이아웃 스래싱 최소화
  const optimizeLayout = () => {
    if (boxRef.current) {
      // 1. 모든 DOM 읽기 작업을 먼저 수행
      const currentWidth = boxRef.current.clientWidth;
      console.log(`✨ Optimized: Initial Width: ${currentWidth}`);

      // 2. 다음 애니메이션 프레임에서 DOM 쓰기 작업을 예약
      requestAnimationFrame(() => {
        if (boxRef.current) {
          boxRef.current.style.height = `${Math.random() * 50 + 50}px`;
          boxRef.current.style.width = `${currentWidth + 10}px`; // 읽은 값을 활용한 쓰기
        }
      });
    }
  };

  return (
    <div style={{ padding: '20px' }}>
      <h3>레이아웃 스래싱 예시</h3>
      <div
        ref={boxRef}
        style={{
          width: '100px',
          height: '50px',
          backgroundColor: '#FFDDC1',
          margin: '10px',
          transition: 'all 0.3s ease',
        }}
      >
        Box
      </div>
      <button onClick={() => setTrigger(prev => prev + 1)}>
        🔴 레이아웃 스래싱 유발
      </button>
      <button onClick={optimizeLayout} style={{ marginLeft: '10px' }}>
        ✨ 레이아웃 스래싱 최소화 (RAF)
      </button>
      <p>
        개발자 도구의 Performance 탭에서 'Layout' 이벤트 발생 빈도를 비교해보세요.
      </p>
    </div>
  );
};

export default LayoutThrashingExample;
```

**실무 적용 팁:**
*   DOM 읽기(예: `clientWidth`, `getBoundingClientRect()`)와 쓰기(예: `style.width`, `transform`) 작업을 분리하고, 가능한 한 쓰기 작업을 `requestAnimationFrame` 내에서 배치 처리하세요.
*   React에서는 직접적인 DOM 조작을 최소화하고, 상태(state)를 통해 렌더링을 유도하는 방식을 선호해야 합니다. 하지만 불가피하게 DOM을 직접 다룰 때는 위 예시처럼 주의해야 합니다.

### 2. INP (Interaction to Next Paint)

**핵심 개념:** INP는 사용자의 첫 상호작용(클릭, 탭, 키보드 입력 등)부터 브라우저가 다음 시각적 업데이트(페인트)를 수행하기까지의 지연 시간을 측정하는 새로운 핵심 웹 바이탈 지표입니다. 이는 웹 페이지의 전반적인 반응성을 나타내며, 특히 복잡한 상호작용이 많은 Single Page Application(SPA)에서 중요하게 작용합니다. 낮은 INP 점수는 사용자가 웹사이트를 "느리다"고 느끼게 하는 주된 원인이 됩니다.

**실무 적용 팁:**
*   **긴 작업 분할:** 사용자 상호작용으로 인해 발생하는 JavaScript 작업을 작은 조각으로 나누어 메인 스레드가 오랫동안 블로킹되지 않도록 합니다. `setTimeout`이나 `requestIdleCallback`을 활용할 수 있습니다.
*   **이벤트 핸들러 최적화:** `debounce`나 `throttle`을 사용하여 이벤트 핸들러의 실행 빈도를 제한합니다.
*   **렌더링 최적화:** React의 `React.memo`, `useCallback`, `useMemo`를 적절히 사용하여 불필요한 리렌더링을 방지하고 컴포넌트 업데이트 비용을 줄입니다.
*   **Next.js 환경:** Next.js의 SSR/SSG 기능을 활용하여 초기 로딩 성능을 개선하고, `next/dynamic`을 이용한 코드 스플리팅으로 필요한 코드만 로드하여 JS 번들 크기를 줄입니다.

### 3. `will-change` 속성

**핵심 개념:** `will-change` CSS 속성은 브라우저에게 특정 요소의 어떤 속성이 곧 변경될 것임을 미리 알려 최적화를 준비하게 합니다. 예를 들어, `will-change: transform;`을 지정하면 브라우저는 해당 요소를 별도의 레이어로 승격시켜 GPU 가속을 활용할 준비를 합니다. 이는 애니메이션이나 트랜지션이 시작될 때 발생할 수 있는 버벅거림을 줄여줍니다. 단, 너무 많은 요소에 남용하면 오히려 메모리 사용량을 늘려 성능 저하를 일으킬 수 있으므로 주의해야 합니다.

**실제 코드 예시 (React/TypeScript + CSS Modules):**

```tsx
// components/WillChangeExample.tsx
import React, { useState } from 'react';
import styles from './WillChangeExample.module.css'; // CSS Modules 사용 예시

const WillChangeExample: React.FC = () => {
  const [isActive, setIsActive] = useState(false);

  return (
    <div style={{ padding: '20px' }}>
      <h3>`will-change` 속성 활용</h3>
      <div
        className={`${styles.animatedElement} ${isActive ? styles.active : ''}`}
        onClick={() => setIsActive(!isActive)}
      >
        {isActive ? 'Active!' : 'Click Me!'}
      </div>
      <p>
        `will-change: transform;` 속성을 통해 애니메이션 시작 전 브라우저가 최적화를 수행할 수 있도록 돕습니다.
        개발자 도구의 'Layers' 탭에서 요소의 레이어 승격 여부를 확인할 수 있습니다.
      </p>
    </div>
  );
};

export default WillChangeExample;
```

```css
/* components/WillChangeExample.module.css */
.animatedElement {
  width: 150px;
  height: 100px;
  background-color: #6C7A89;
  color: white;
  display: flex;
  align-items: center;
  justify-content: center;
  margin: 20px 0;
  cursor: pointer;
  transition: transform 0.5s ease-out;
}

.active {
  transform: translateX(100px) rotate(15deg) scale(1.2);
  will-change: transform; /* ✨ 이 요소의 transform 속성이 변경될 것임을 브라우저에 미리 알림 */
}
```

**실무 적용 팁:**
*   애니메이션이나 트랜지션이 시작되기 전에만 적용하고, 애니메이션이 끝난 후에는 제거하는 것이 이상적입니다. (예: JS로 클래스 토글)
*   움직임이 복잡하거나 성능에 민감한 요소에만 신중하게 적용합니다.

### 4. 패시브 리스너 (Passive Listeners)

**핵심 개념:** `touchstart`, `touchmove`, `wheel`, `scroll`과 같은 이벤트들은 기본적으로 동기적으로 처리되어 메인 스레드를 블로킹할 수 있습니다. 특히 `e.preventDefault()`를 호출하여 스크롤 동작을 막는 경우, 브라우저는 해당 이벤트 리스너가 끝날 때까지 스크롤을 시작할지 말지 결정할 수 없어 지연이 발생합니다. `passive: true` 옵션을 사용하면 이벤트 리스너가 `preventDefault()`를 호출하지 않을 것임을 브라우저에 알려, 브라우저가 이벤트 리스너의 실행을 기다리지 않고 스크롤/터치 동작을 즉시 시작할 수 있도록 합니다. 이는 스크롤 성능을 크게 향상시킵니다.

**실제 코드 예시 (JavaScript/TypeScript):**

```typescript
// utils/eventListeners.ts 또는 최상위 컴포넌트의 useEffect
import { useEffect } from 'react';

const usePassiveScrollListener = () => {
  useEffect(() => {
    const handleScroll = (e: Event) => {
      // 스크롤 이벤트 처리 로직 (여기서는 e.preventDefault()를 호출하지 않습니다.)
      console.log('Scrolling...', e.timeStamp);
    };

    // ✨ passive: true 옵션을 추가하여 스크롤 성능 향상
    window.addEventListener('scroll', handleScroll, { passive: true });
    // window.addEventListener('touchstart', handleScroll, { passive: true });
    // window.addEventListener('touchmove', handleScroll, { passive: true });

    return () => {
      window.removeEventListener('scroll', handleScroll, { passive: true });
      // window.removeEventListener('touchstart', handleScroll, { passive: true });
      // window.removeEventListener('touchmove', handleScroll, { passive: true });
    };
  }, []);
};

export default usePassiveScrollListener;

// Next.js 페이지 또는 _app.tsx 에서 사용 예시
// import usePassiveScrollListener from '../utils/eventListeners';
// const MyPage: React.FC = () => {
//   usePassiveScrollListener();
//   // ...
// }
```

**실무 적용 팁:**
*   스크롤, 터치 관련 이벤트 리스너를 추가할 때 `passive: true` 옵션을 적극적으로 활용하세요.
*   `e.preventDefault()`를 호출해야 하는 경우(예: 커스텀 스크롤바 구현)에는 `passive: false`를 명시적으로 지정하거나 옵션을 생략해야 합니다. (기본값은 `passive: false`와 유사하게 동작합니다.)
*   이벤트 리스너가 많아질수록 성능에 미치는 영향이 커지므로, 필요한 곳에만 최소한으로 적용하는 것이 중요합니다.

## 🛠️ 실무 적용 팁: 성능 최적화를 위한 종합 전략

위에서 다룬 핵심 개념 외에도 프론트엔드 성능을 향상시키기 위한 다양한 실무 전략들이 있습니다.

1.  **측정, 또 측정! (Measure, Don't Guess):**
    *   **Chrome 개발자 도구:** Lighthouse, Performance 탭, Network 탭을 활용하여 렌더링 과정, 네트워크 요청, JS 실행 시간 등을 분석합니다.
    *   **Web Vitals:** Google Search Console, PageSpeed Insights, Web Vitals Chrome Extension 등을 통해 실제 사용자 경험(RUM: Real User Monitoring) 지표를 모니터링합니다. 특히 INP는 사용자 상호작용에 초점을 맞추므로 면밀히 관찰해야 합니다.
    *   **Next.js Analytics:** Vercel 배포 시 제공되는 Web Vitals 대시보드를 적극 활용하여 실제 사용자 데이터를 기반으로 성능을 개선합니다.

2.  **코드 스플리팅 및 지연 로딩 (Code Splitting & Lazy Loading):**
    *   `React.lazy()`와 `Suspense`를 사용하여 컴포넌트를 필요할 때만 로드합니다.
    *   Next.js에서는 `next/dynamic`을 통해 자동으로 코드 스플리팅을 적용할 수 있습니다.
    *   라우트 기반 코드 스플리팅을 통해 초기 로드 번들 크기를 최소화합니다.

3.  **이미지 및 미디어 최적화:**
    *   `next/image` 컴포넌트를 사용하여 이미지 크기 최적화, 지연 로딩, 최신 이미지 포맷(WebP, AVIF) 자동 변환을 적용합니다.
    *   적절한 이미지 크기와 해상도를 사용하고, CDN을 활용하여 빠르게 전송합니다.

4.  **데이터 페칭 최적화:**
    *   Next.js의 `getServerSideProps`, `getStaticProps`, `getStaticPaths`를 활용하여 서버에서 데이터를 미리 가져오거나 빌드 시점에 생성하여 초기 로딩 속도를 향상시킵니다.
    *   클라이언트 사이드 페칭 시에는 SWR, React Query와 같은 라이브러리를 사용하여 캐싱, 재검증, 중복 요청 방지 등의 기능을 활용합니다.

5.  **번들 크기 최적화:**
    *   불필요한 라이브러리 제거, 트리 쉐이킹(Tree Shaking)을 통해 최종 번들 크기를 줄입니다.
    *   Webpack Bundle Analyzer와 같은 도구를 사용하여 번들의 구성을 분석하고 최적화 기회를 찾습니다.

6.  **CSS 최적화:**
    *   CSS-in-JS 라이브러리 사용 시, 런타임 오버헤드를 고려하고, 빌드 시점에 CSS를 추출하는 옵션을 활용합니다.
    *   Tailwind CSS나 CSS Modules와 같이 컴포넌트 기반으로 격리된 CSS를 사용하여 불필요한 스타일이 로드되는 것을 방지합니다.

7.  **애니메이션 최적화:**
    *   성능에 부담이 적은 `transform`, `opacity` 속성을 주로 사용하고, `will-change`를 적절히 활용합니다.
    *   가능하다면 CSS 애니메이션을 사용하고, 복잡한 JS 기반 애니메이션은 `requestAnimationFrame`을 통해 구현합니다.

## 🎯 마무리: 지속적인 관심과 개선이 핵심

오늘 우리는 프론트엔드 성능 최적화의 핵심 개념인 레이아웃 스래싱, INP, `will-change`, 패시브 리스너를 살펴보고, React/Next.js/TypeScript 환경에서의 실무 적용 팁을 알아보았습니다. 웹 기술이 발전함에 따라 사용자들의 기대치 또한 높아지고 있으며, 이제 성능은 단순한 기술적 과제를 넘어 비즈니스 성공의 필수 요소가 되었습니다.

성능 최적화는 한 번의 작업으로 끝나는 것이 아니라, 지속적인 측정, 분석, 개선의 과정입니다. 개발자 도구와 Web Vitals 지표를 꾸준히 모니터링하며, 새로운 기술과 패턴을 학습하고 적용하는 노력이 필요합니다. 사용자에게 더 빠르고 만족스러운 경험을 제공하기 위해, 우리 모두 '체감 속도'를 디자인하는 프론트엔드 개발자가 되도록 노력합시다!

다음 학습 방향으로는 WebAssembly, Service Worker를 활용한 오프라인 경험 개선, 그리고 서버 컴포넌트(Server Components)와 같은 새로운 렌더링 패러다임이 성능에 미치는 영향에 대해 더 깊이 탐구해보는 것을 추천합니다.
