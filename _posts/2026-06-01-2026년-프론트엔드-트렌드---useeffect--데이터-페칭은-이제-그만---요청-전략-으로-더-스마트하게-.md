오늘 날짜는 **2026년 6월 1일**입니다. 시간이 정말 빠르죠? 2026년의 프론트엔드 개발은 빠르게 진화하고 있으며, 과거의 방식에만 머물러서는 도태되기 쉽습니다. 특히 데이터 페칭(Data Fetching)과 관련해서는 지난 몇 년간 패러다임의 큰 변화가 있었습니다.

오늘은 2026년에도 여전히 많은 개발자들이 고민하는 `useEffect` 기반 데이터 페칭의 문제점과, 이를 대체할 더 효율적이고 강력한 '요청 전략(Request Strategies)'에 대해 깊이 파고들어 보겠습니다.

## 🚀 소개: `useEffect`의 한계, 그리고 새로운 지평

React 개발을 시작할 때, 많은 분들이 `useEffect` 훅을 데이터 페칭의 만능 해결사로 여겼을 겁니다. 저 또한 그랬습니다. 컴포넌트 마운트 시 데이터를 가져오고, 특정 의존성(dependency)이 변경될 때 다시 데이터를 불러오는 것이 자연스럽게 느껴졌죠.

하지만 시간이 지나고 프로젝트 규모가 커지면서, `useEffect`를 이용한 데이터 페칭은 수많은 골칫거리를 안겨주기 시작했습니다. 불필요한 재요청, 복잡한 로딩/에러 상태 관리, 경쟁 조건(race condition) 문제, 그리고 무엇보다도 낮은 개발자 경험(DX)은 우리를 지치게 만들었습니다. 2026년, 우리는 이제 `useEffect`를 데이터 페칭의 주력으로 사용하던 시대에서 벗어나, 더욱 견고하고 효율적인 '요청 전략'을 모색해야 할 때입니다.

## 💡 본문: 더 나은 데이터 페칭을 위한 '요청 전략'

### 핵심 개념 설명: `useEffect`의 늪에서 벗어나기

`useEffect`는 컴포넌트의 생명주기와 동기화된 부수 효과(side effect)를 처리하는 데 강력한 도구입니다. DOM 직접 조작, 이벤트 리스너 등록/해제, 외부 시스템과의 동기화 등이 그 예시죠. 하지만 데이터 페칭은 단순히 "부수 효과"를 넘어, 네트워크 상태, 캐싱, 재시도, 데이터 무효화 등 복잡한 비동기 로직과 상태 관리를 필요로 합니다.

`useEffect`로 이 모든 것을 처리하려면 다음과 같은 문제에 직면합니다.

1.  **복잡한 상태 관리:** `isLoading`, `isError`, `data` 등의 상태를 직접 관리해야 하며, 비동기 로직의 특성상 `isMounted` 플래그를 이용한 클린업 로직까지 추가해야 합니다.
2.  **경쟁 조건:** 빠르게 컴포넌트가 언마운트되거나 의존성이 변경될 때, 이전 요청의 응답이 나중에 도착하여 잘못된 상태를 업데이트할 수 있습니다.
3.  **캐싱 부재:** 매번 새로운 요청을 보내므로, 동일한 데이터를 여러 컴포넌트에서 필요로 할 때 비효율적인 네트워크 요청이 발생합니다.
4.  **보일러플레이트 코드:** 각 컴포넌트마다 거의 동일한 페칭 로직을 반복해서 작성하게 됩니다.
5.  **낮은 개발자 경험:** 위 문제들을 해결하기 위해 코드가 점점 복잡해지고 디버깅이 어려워집니다.

이러한 문제들을 해결하기 위해 등장한 것이 바로 **'요청 전략(Request Strategies)'**입니다. 이는 데이터 페칭과 관련된 모든 복잡한 로직(네트워크 요청, 캐싱, 재시도, 데이터 동기화, 에러 처리, 로딩 상태 등)을 추상화하고, 선언적인 방식으로 데이터를 관리할 수 있도록 돕는 패턴 또는 라이브러리를 의미합니다. 대표적으로 TanStack Query (구 React Query), SWR, Apollo Client (GraphQL) 등이 있습니다.

이러한 라이브러리들은 다음과 같은 이점을 제공합니다.

*   **선언적 데이터 페칭:** `useEffect` 내부의 명령형 로직 대신, 어떤 데이터를 가져올지 선언하기만 하면 됩니다.
*   **자동 캐싱 및 재검증:** 한 번 가져온 데이터는 캐시하고, 필요에 따라 자동으로 최신 상태로 재검증(revalidation)하여 성능을 최적화합니다.
*   **간편한 상태 관리:** `isLoading`, `isError`, `data` 등의 상태를 훅 형태로 제공하여 쉽게 접근할 수 있습니다.
*   **백그라운드 페칭:** 사용자에게 끊김 없는 경험을 제공하기 위해 백그라운드에서 데이터를 업데이트합니다.
*   **자동 재시도 및 에러 처리:** 네트워크 오류 발생 시 자동으로 재시도하고, 에러 상태를 효과적으로 관리합니다.
*   **옵티미스틱 업데이트:** 사용자 경험을 향상시키기 위해 서버 응답 전에 UI를 미리 업데이트할 수 있습니다.

### 실제 코드 예시: `useEffect`에서 벗어나기

#### 🙅‍♂️ `useEffect`를 이용한 데이터 페칭 (구식 방식)

```typescript
import React, { useState, useEffect } from 'react';

interface Post {
  id: number;
  title: string;
  body: string;
}

function OldSchoolPostFetcher({ postId }: { postId: number }) {
  const [post, setPost] = useState<Post | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    let isMounted = true; // 언마운트된 컴포넌트에서 상태 업데이트 방지
    const abortController = new AbortController(); // 요청 취소를 위한 컨트롤러

    const fetchPost = async () => {
      try {
        setLoading(true);
        setError(null);
        const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${postId}`, {
          signal: abortController.signal, // 요청 취소 시그널 연결
        });

        if (!response.ok) {
          throw new Error(`HTTP error! status: ${response.status}`);
        }
        const data: Post = await response.json();

        if (isMounted) {
          setPost(data);
        }
      } catch (err) {
        if (err instanceof DOMException && err.name === 'AbortError') {
          console.log('Fetch aborted');
          return; // 요청 취소는 에러로 처리하지 않음
        }
        if (isMounted) {
          setError(err instanceof Error ? err.message : String(err));
        }
      } finally {
        if (isMounted) {
          setLoading(false);
        }
      }
    };

    fetchPost();

    return () => {
      isMounted = false;
      abortController.abort(); // 컴포넌트 언마운트 또는 의존성 변경 시 요청 취소
    };
  }, [postId]); // postId가 변경될 때마다 재실행

  if (loading) return <p>게시물 로딩 중...</p>;
  if (error) return <p className="text-red-500">오류 발생: {error}</p>;
  if (!post) return <p>게시물을 찾을 수 없습니다.</p>;

  return (
    <div className="p-4 border rounded shadow-md">
      <h3 className="text-xl font-bold mb-2">{post.title}</h3>
      <p className="text-gray-700">{post.body}</p>
    </div>
  );
}
```
위 코드는 `useEffect`의 문제점을 해결하기 위해 `isMounted` 플래그와 `AbortController`까지 동원했지만, 여전히 복잡하고 에러 핸들링, 캐싱 등은 직접 구현해야 합니다.

#### 🙆‍♂️ '요청 전략'을 이용한 데이터 페칭 (현대적인 방식)

이제 TanStack Query (React Query)를 사용하여 동일한 기능을 구현해봅시다. Next.js 환경에서 TypeScript와 함께 사용한다고 가정합니다.

```typescript
// src/services/postService.ts
// 데이터 페칭 로직을 별도로 분리하여 재사용성 및 가독성 향상
interface Post {
  id: number;
  title: string;
  body: string;
}

export async function fetchPostById(id: number): Promise<Post> {
  const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
  if (!response.ok) {
    throw new Error(`HTTP error! status: ${response.status}`);
  }
  return response.json();
}
```

```typescript
// src/components/NewSchoolPostFetcher.tsx
import React from 'react';
import { useQuery } from '@tanstack/react-query'; // TanStack Query 훅

interface Post {
  id: number;
  title: string;
  body: string;
}

function NewSchoolPostFetcher({ postId }: { postId: number }) {
  // useQuery 훅을 사용하여 데이터를 페칭하고 상태를 관리합니다.
  const { 
    data: post,        // 페칭된 데이터 (Post 타입)
    isLoading,         // 로딩 중인지 여부
    isError,           // 에러 발생 여부
    error,             // 에러 객체
    isFetching,        // 백그라운드에서 페칭 중인지 여부
    refetch            // 수동으로 데이터를 다시 가져오는 함수
  } = useQuery<Post, Error>({
    queryKey: ['post', postId], // 이 쿼리를 식별하는 고유 키. postId가 변경되면 새 쿼리로 간주
    queryFn: () => fetchPostById(postId), // 실제 데이터 페칭 함수
    staleTime: 5 * 60 * 1000, // 5분 동안 "fresh" 상태 유지 (이 시간 동안은 재요청 없음)
    cacheTime: 10 * 60 * 1000, // 10분 동안 캐시된 데이터 유지 (이후 가비지 컬렉션 대상)
    enabled: !!postId, // postId가 있을 때만 쿼리 실행
  });

  if (isLoading) return <p>게시물 로딩 중...</p>;
  if (isError) return <p className="text-red-500">오류 발생: {error?.message}</p>;
  if (!post) return <p>게시물을 찾을 수 없습니다.</p>; // 데이터가 아직 없을 경우

  return (
    <div className="p-4 border rounded shadow-md">
      <h3 className="text-xl font-bold mb-2">{post.title}</h3>
      <p className="text-gray-700">{post.body}</p>
      {isFetching && <p className="text-sm text-gray-500">백그라운드에서 업데이트 중...</p>}
      <button 
        onClick={() => refetch()} 
        className="mt-4 px-4 py-2 bg-blue-500 text-white rounded hover:bg-blue-600"
      >
        새로고침
      </button>
    </div>
  );
}
```

`useQuery` 훅을 사용하니 코드가 훨씬 간결해지고, 캐싱, 재시도, 백그라운드 업데이트 등 복잡한 로직을 직접 구현할 필요가 없어졌습니다. `isLoading`, `isError`, `data` 같은 상태도 직관적으로 제공되어 개발자 경험이 크게 향상됩니다.

### 실무 적용 팁: 2026년의 데이터 페칭 전략

1.  **`useEffect`는 부수 효과에 집중하세요:**
    *   DOM 직접 조작 (스크롤 이벤트, 캔버스 드로잉)
    *   이벤트 리스너 등록 및 해제
    *   외부 라이브러리 (애니메이션, 차트) 초기화 및 정리
    *   웹소켓 연결 및 해제
    *   폼 유효성 검사 로직 (데이터 페칭이 아닌 순수 클라이언트 로직)
    데이터 페칭은 `useEffect`의 주된 용도가 아닙니다.

2.  **프로젝트에 맞는 요청 전략 라이브러리 선택:**
    *   **TanStack Query (React Query):** 강력한 캐싱, 재시도, 옵티미스틱 업데이트, DevTools 등 비동기 데이터 관리에 필요한 거의 모든 기능을 제공합니다. 대부분의 React/Next.js 프로젝트에 적합합니다.
    *   **SWR:** Vercel에서 개발했으며, TanStack Query와 유사한 기능을 제공하지만 좀 더 가볍고 설정이 간단합니다. Next.js 프로젝트에 잘 어울립니다.
    *   **Apollo Client/Relay (GraphQL):** GraphQL API를 사용하는 경우, 데이터 페칭과 상태 관리를 통합하여 강력한 기능을 제공합니다.

3.  **데이터 무효화 (Invalidation) 전략:**
    *   데이터 변경(생성, 수정, 삭제) 후에는 관련 쿼리를 무효화하여 최신 데이터를 다시 가져오도록 해야 합니다. `queryClient.invalidateQueries(['post', postId])`와 같은 기능을 활용하세요.
    *   `onSuccess` 콜백을 사용하여 특정 액션 성공 시 자동으로 쿼리를 무효화하는 패턴을 구축하면 편리합니다.

4.  **Next.js 환경에서의 서버 사이드 페칭:**
    *   `getServerSideProps`나 `getStaticProps`에서 TanStack Query/SWR의 `prefetchQuery` 또는 `preload` 기능을 사용하여 초기 데이터를 서버에서 미리 가져오고, 클라이언트에서 하이드레이션(hydration)하여 사용자 경험을 최적화할 수 있습니다.
    *   Next.js 13+의 App Router 환경에서는 React Server Components(RSC)를 활용한 데이터 페칭이 더욱 중요해지고 있습니다. 이는 서버에서 데이터를 직접 페칭하여 클라이언트로 전송하므로, 클라이언트 측 `useEffect`의 필요성을 더욱 줄여줍니다.

5.  **에러 핸들링 및 로딩 UI 개선:**
    *   `isError`, `error` 상태를 이용하여 사용자에게 친화적인 에러 메시지를 제공하세요.
    *   `isLoading`, `isFetching` 상태를 구분하여 초기 로딩과 백그라운드 재검증을 시각적으로 다르게 처리하면 사용자 경험을 향상시킬 수 있습니다. (예: 스켈레톤 UI, 스피너)

## 🎯 마무리: 더 나은 개발, 더 나은 사용자 경험

2026년의 프론트엔드 개발은 `useEffect`의 복잡한 굴레에서 벗어나, 데이터 페칭을 위한 전문적인 '요청 전략'을 적극적으로 도입하는 방향으로 나아가고 있습니다. 이는 단순히 코드를 줄이는 것을 넘어, 애플리케이션의 성능, 안정성, 그리고 무엇보다 개발자 경험을 혁신적으로 개선하는 길입니다.

아직 `useEffect`로 데이터 페칭을 하고 있다면, 이제는 과감히 새로운 패러다임으로 전환할 때입니다. TanStack Query나 SWR 같은 라이브러리를 학습하고 프로젝트에 적용해보세요. 처음에는 낯설 수 있지만, 일단 익숙해지면 훨씬 더 즐겁고 효율적인 개발을 경험하게 될 것입니다.

다음번에는 Next.js 14+와 React Server Components 환경에서 데이터 페칭 전략을 어떻게 더욱 고도화할 수 있을지에 대해 다뤄보겠습니다.
