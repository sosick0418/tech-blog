안녕하세요, 프론트엔드 개발 전문 블로거 개발하는 진저입니다. 2026년 6월 8일, 오늘도 프론트엔드의 뜨거운 트렌드를 함께 살펴보겠습니다.

오늘 다룰 주제는 바로 '클린 아키텍처'입니다. 프론트엔드 개발이 점점 더 복잡해지면서, 단순히 UI를 렌더링하는 것을 넘어 백엔드 못지않은 비즈니스 로직을 다루는 경우가 많아졌습니다. 이로 인해 코드의 유지보수성과 확장성이 저하되는 문제에 직면하곤 하죠.

이번 포스팅에서는 이러한 문제에 대한 해답으로 제시되는 '프론트엔드 클린 아키텍처'에 대해 깊이 파고들어 보겠습니다.



## 소개

2026년 6월, 프론트엔드 개발은 더 이상 단순한 화면 구현을 넘어섭니다. 사용자 경험의 복잡성은 증가하고, 실시간 데이터 처리, 다양한 상태 관리, 그리고 때로는 백엔드의 핵심 비즈니스 로직과 밀접하게 연관된 프론트엔드 로직까지 다루게 되죠. 이러한 환경에서 컴포넌트에 UI 로직과 비즈니스 로직이 뒤섞여 버리면, 코드는 빠르게 '스파게티'가 되고 유지보수와 확장은 악몽으로 변합니다. 오늘 우리는 이 문제에 대한 강력한 해결책인 '프론트엔드 클린 아키텍처'를 탐구하고, 왜 이 접근 방식이 여러분의 프로젝트에 필수적인지 알아볼 것입니다.

## 본문

### 핵심 개념 설명: 프론트엔드 클린 아키텍처란?

클린 아키텍처는 원래 백엔드 시스템의 복잡성을 관리하기 위해 제안된 아키텍처 패턴입니다. 핵심은 '관심사의 분리(Separation of Concerns)'와 '의존성 역전 원칙(Dependency Inversion Principle)'을 통해 시스템의 계층을 명확히 나누고, 각 계층이 독립적으로 발전할 수 있도록 하는 것입니다. 프론트엔드에 이를 적용하면 다음과 같은 계층으로 나눌 수 있습니다:

1.  **Domain (도메인):** 시스템의 핵심 비즈니스 규칙과 엔티티를 정의합니다. 특정 UI 프레임워크나 데이터베이스에 의존하지 않는 순수한 비즈니스 로직을 담습니다. 예를 들어, '상품'이나 '주문'과 같은 데이터 구조와 그 데이터를 조작하는 순수 함수들이 여기에 속합니다.
2.  **Use Cases (유스케이스 / Application):** 도메인 계층의 엔티티를 사용하여 특정 비즈니스 흐름을 구현합니다. 사용자의 특정 액션(예: "상품을 장바구니에 담기", "결제하기")에 대한 로직을 처리합니다. 이 계층 또한 UI나 외부 시스템에 대한 직접적인 의존성을 갖지 않습니다.
3.  **Presentation (표현 / UI):** 사용자에게 보여지는 UI와 사용자 입력을 처리하는 부분을 담당합니다. React 컴포넌트, 훅, 컨텍스트 등이 여기에 해당하며, 유스케이스 계층을 호출하여 비즈니스 로직을 실행하고, 도메인 계층의 데이터를 받아 UI를 렌더링합니다. 이 계층은 다른 계층에 의존하지만, 다른 계층은 이 계층에 의존하지 않습니다.

이러한 분리를 통해, 비즈니스 로직은 UI와 독립적으로 테스트되고 변경될 수 있으며, UI는 비즈니스 로직에 대한 걱정 없이 오직 렌더링에만 집중할 수 있게 됩니다. 이는 단순히 '스마트/덤 컴포넌트'를 나누는 것을 넘어, 비즈니스 '의미' 자체를 UI '메커니즘'으로부터 완전히 분리하는 것을 목표로 합니다.

또한, 컴포넌트를 설계할 때 "API처럼 설계하라"는 원칙은 클린 아키텍처의 Presentation 계층에서 매우 중요합니다. 컴포넌트가 마치 잘 정의된 API처럼 명확한 Props 인터페이스와 예측 가능한 동작을 가지도록 설계하면, 재사용성, 테스트 용이성, 그리고 코드 가독성이 극대화됩니다. 이는 컴포넌트가 내부 비즈니스 로직을 숨기고, 외부에는 필요한 기능만을 노출하는 캡슐화와도 연결됩니다.

### 실제 코드 예시 (React/Next.js/TypeScript 활용)

인보이스(송장) 앱을 예시로 들어보겠습니다. 기존에는 `InvoiceItem` 컴포넌트 내부에 상품 가격 계산 로직이 직접 포함되어 있었을 것입니다.

**기존 방식 (비즈니스 로직과 UI가 혼재):**

```tsx
// components/InvoiceItem.tsx
import React, { useState } from 'react';

interface InvoiceItemProps {
  id: string;
  name: string;
  unitPrice: number;
  initialQuantity: number;
  onUpdateTotal: (itemId: string, newTotal: number) => void;
}

const InvoiceItem: React.FC<InvoiceItemProps> = ({
  id,
  name,
  unitPrice,
  initialQuantity,
  onUpdateTotal,
}) => {
  const [quantity, setQuantity] = useState(initialQuantity);

  // ❌ 비즈니스 로직이 컴포넌트 내부에 직접 존재
  const itemTotal = unitPrice * quantity;

  const handleQuantityChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const newQuantity = parseInt(e.target.value) || 0;
    setQuantity(newQuantity);
    // ❌ 총액 계산 로직이 UI 이벤트 핸들러와 섞여 있음
    onUpdateTotal(id, unitPrice * newQuantity);
  };

  return (
    <div>
      <span>{name}</span>
      <span>가격: {unitPrice.toLocaleString()}원</span>
      <input type="number" value={quantity} onChange={handleQuantityChange} min="0" />
      <span>총액: {itemTotal.toLocaleString()}원</span>
    </div>
  );
};

export default InvoiceItem;
```

**클린 아키텍처 적용 (비즈니스 로직 분리):**

이제 비즈니스 로직을 `domain`과 `usecases` 계층으로 분리하고, `presentation` 계층은 이들을 활용하도록 변경해봅시다.

**1. Domain 계층 (`src/domain/invoice.ts`)**
순수한 비즈니스 엔티티와 핵심 로직을 정의합니다.

```typescript
// src/domain/invoice.ts

export interface InvoiceItem {
  id: string;
  name: string;
  unitPrice: number;
  quantity: number;
}

export interface Invoice {
  id: string;
  customerName: string;
  items: InvoiceItem[];
}

// 순수한 비즈니스 로직: 특정 아이템의 총액 계산
export const calculateItemTotal = (item: InvoiceItem): number => {
  return item.unitPrice * item.quantity;
};

// 순수한 비즈니스 로직: 인보이스 전체 총액 계산
export const calculateInvoiceGrandTotal = (invoice: Invoice): number => {
  return invoice.items.reduce((acc, item) => acc + calculateItemTotal(item), 0);
};
```

**2. Use Cases 계층 (`src/usecases/updateInvoiceItemQuantity.ts`)**
도메인 로직을 사용하여 특정 비즈니스 시나리오를 처리합니다.

```typescript
// src/usecases/updateInvoiceItemQuantity.ts
import { Invoice, InvoiceItem } from '../domain/invoice';

/**
 * 인보이스 아이템의 수량을 업데이트하는 유스케이스
 * @param invoice - 현재 인보이스 객체
 * @param itemId - 업데이트할 아이템 ID
 * @param newQuantity - 새로운 수량 (0 이상)
 * @returns 업데이트된 새로운 인보이스 객체
 */
export const updateInvoiceItemQuantity = (
  invoice: Invoice,
  itemId: string,
  newQuantity: number
): Invoice => {
  const validatedQuantity = Math.max(0, newQuantity); // 수량은 0 미만이 될 수 없음

  const updatedItems = invoice.items.map(item =>
    item.id === itemId ? { ...item, quantity: validatedQuantity } : item
  );

  return { ...invoice, items: updatedItems };
};
```

**3. Presentation 계층 (`src/presentation/hooks/useInvoiceManager.ts`, `src/presentation/components/InvoiceItemRow.tsx`, `src/presentation/components/InvoiceDetail.tsx`)**
UI 렌더링과 사용자 상호작용을 담당하며, 유스케이스를 호출합니다.

```typescript
// src/presentation/hooks/useInvoiceManager.ts (React Custom Hook)
import { useState, useCallback } from 'react';
import { Invoice, calculateInvoiceGrandTotal } from '../../domain/invoice';
import { updateInvoiceItemQuantity } from '../../usecases/updateInvoiceItemQuantity';

export const useInvoiceManager = (initialInvoice: Invoice) => {
  const [invoice, setInvoice] = useState<Invoice>(initialInvoice);

  const handleUpdateItemQuantity = useCallback((itemId: string, newQuantity: number) => {
    // 유스케이스를 호출하여 비즈니스 로직 실행
    setInvoice(prevInvoice => updateInvoiceItemQuantity(prevInvoice, itemId, newQuantity));
  }, []);

  // 도메인 로직을 사용하여 총액 계산 (UI에 표시하기 위함)
  const grandTotal = calculateInvoiceGrandTotal(invoice);

  return {
    invoice,
    grandTotal,
    handleUpdateItemQuantity,
  };
};
```

```tsx
// src/presentation/components/InvoiceItemRow.tsx (UI 렌더링 전용 컴포넌트)
import React from 'react';
import { InvoiceItem } from '../../domain/invoice';

interface InvoiceItemRowProps {
  item: InvoiceItem;
  onQuantityChange: (itemId: string, newQuantity: number) => void;
  itemTotal: number; // 계산된 총액을 props로 받음
}

const InvoiceItemRow: React.FC<InvoiceItemRowProps> = ({ item, onQuantityChange, itemTotal }) => {
  return (
    <tr>
      <td>{item.name}</td>
      <td>{item.unitPrice.toLocaleString()}</td>
      <td>
        <input
          type="number"
          value={item.quantity}
          onChange={(e) => onQuantityChange(item.id, parseInt(e.target.value) || 0)}
          min="0"
        />
      </td>
      <td>{itemTotal.toLocaleString()}원</td>
    </tr>
  );
};

export default InvoiceItemRow;
```

```tsx
// src/presentation/components/InvoiceDetail.tsx (페이지/컨테이너 컴포넌트)
import React from 'react';
import { useInvoiceManager } from '../hooks/useInvoiceManager';
import InvoiceItemRow from './InvoiceItemRow';
import { calculateItemTotal } from '../../domain/invoice'; // UI 표시를 위해 도메인 로직 사용

// 예시 초기 데이터
const initialInvoiceData = {
  id: 'INV-2026-001',
  customerName: '프론트엔드 주식회사',
  items: [
    { id: '1', name: 'React 서적', unitPrice: 35000, quantity: 1 },
    { id: '2', name: 'TypeScript 강의', unitPrice: 120000, quantity: 2 },
  ],
};

const InvoiceDetail: React.FC = () => {
  const { invoice, grandTotal, handleUpdateItemQuantity } = useInvoiceManager(initialInvoiceData);

  return (
    <div>
      <h1>Invoice #{invoice.id}</h1>
      <p>고객명: {invoice.customerName}</p>
      <table>
        <thead>
          <tr>
            <th>상품명</th>
            <th>단가</th>
            <th>수량</th>
            <th>총액</th>
          </tr>
        </thead>
        <tbody>
          {invoice.items.map((item) => (
            <InvoiceItemRow
              key={item.id}
              item={item}
              onQuantityChange={handleUpdateItemQuantity}
              itemTotal={calculateItemTotal(item)} // UI 표시를 위한 아이템별 총액 계산
            />
          ))}
        </tbody>
      </table>
      <h2>총 결제 금액: {grandTotal.toLocaleString()}원</h2>
    </div>
  );
};

export default InvoiceDetail;
```

이 예시에서 `InvoiceItemRow`는 오직 `item` 데이터를 받아 렌더링하고, `onQuantityChange` 콜백을 통해 상위 컴포넌트에 변경 사실을 알립니다. 실제 비즈니스 로직(수량 업데이트, 총액 계산)은 `useInvoiceManager` 훅과 `usecases`, `domain` 계층에서 처리됩니다. 이로써 컴포넌트는 UI에만 집중하고, 비즈니스 로직은 재사용 가능하고 테스트하기 쉬운 독립적인 모듈이 됩니다.

### 실무 적용 팁

1.  **점진적 적용:** 모든 것을 한 번에 바꾸려 하지 마세요. 새로운 기능 개발 시 또는 기존의 복잡한 컴포넌트를 리팩토링할 때 클린 아키텍처를 적용해 보세요. 작은 성공 경험이 중요합니다.
2.  **명확한 폴더 구조:** `src/domain`, `src/usecases`, `src/presentation/components`, `src/presentation/hooks` 와 같이 계층별로 명확한 폴더 구조를 가져가세요. 이는 팀원들이 코드의 위치와 역할을 쉽게 파악하는 데 도움을 줍니다.
3.  **팀원과의 합의:** 클린 아키텍처는 팀 전체의 합의와 이해가 필요한 접근 방식입니다. 각 계층의 역할과 의존성 규칙에 대해 팀원들과 충분히 논의하고 가이드라인을 설정하세요.
4.  **테스트 용이성 활용:** 도메인 및 유스케이스 계층은 UI 프레임워크에 의존하지 않으므로, 순수 함수로 구성되어 단위 테스트를 작성하기 매우 용이합니다. 이는 견고한 애플리케이션을 구축하는 데 큰 이점입니다.
5.  **컴포넌트 API 디자인:** `Great Components Are Designed Like APIs` 트렌드에서 강조하듯이, Presentation 계층의 컴포넌트는 명확한 Props 인터페이스를 가지도록 설계하여, 외부에서 컴포넌트의 내부 구현을 몰라도 쉽게 사용할 수 있도록 하세요.

## 마무리

프론트엔드 클린 아키텍처는 초기 학습 곡선이 존재하지만, 애플리케이션의 복잡성이 증가할수록 그 진가가 발휘됩니다. UI와 비즈니스 로직을 명확히 분리함으로써, 우리는 더욱 유연하고, 확장 가능하며, 유지보수하기 쉬운 코드를 작성할 수 있게 됩니다. 이는 단순히 '깔끔한 코드'를 넘어, 미래의 변경에 강건한 시스템을 구축하는 길입니다.

이제 여러분의 프론트엔드 프로젝트에서도 비즈니스 로직을 컴포넌트 너머로 분리하여, 더 견고하고 지속 가능한 아키텍처를 만들어 나가는 여정을 시작해 보세요!

**다음 학습 방향:**
*   **DDD (Domain-Driven Design):** 도메인 계층을 설계하는 데 있어 깊은 통찰을 제공합니다.
*   **TDD (Test-Driven Development):** 비즈니스 로직의 테스트 용이성을 극대화하는 클린 아키텍처와 시너지를 낼 수 있습니다.
*   **Adapter 패턴:** 외부 서비스(API 호출 등)와의 통합을 클린 아키텍처 내에서 어떻게 처리할지 탐구해 보세요.
